Project: “Inbox Autofill” — a production‑grade cross‑browser extension (Chrome/Edge, MV3; later Firefox via WebExtensions) plus a small backend that checks a user’s new Gmail messages daily and helps auto‑fill web forms and PDF forms with the user’s saved profile data.

Primary outcomes

Extension (MV3) that:

Injects a content script to detect form fields (text, email, tel, address, company, EIN/Tax ID, etc.), maps them to user profile keys, and pre‑fills (never auto‑submits) with a visible “Confirm autofill” UI.

Supports on‑page links from emails (CTA buttons that open forms), and attachments (PDF forms) by launching a secure viewer to fill fields and export.

Minimizes permissions: use activeTab + optional host_permissions (request per‑site access at runtime). Provide an allowlist UI.

Local “Profile Vault” (name, address, company, etc.) encrypted at rest in extension storage; easy edit + import/export.

Keyboard shortcut to “Autofill this page”.

Backend (Node 20 + Express) that:

Handles OAuth 2.0 for Gmail with restricted scopes limited to gmail.readonly and label filters.

Uses Gmail Push Notifications + Pub/Sub users.watch and users.history.list to fetch only changes since the last historyId. Store last processed historyId per user.

Extracts actionable items: links to forms, PDF/DOCX attachments, dates/deadlines, sender domain.

Runs URL safety checks (Google Safe Browsing v5 or alternative) before offering autofill suggestions.

Sends the extension a “review queue” of items (via polling endpoint or WebSocket) so users can one‑click open a form and autofill.

PDF forms support

Implement PDF form fill via a JavaScript library that supports AcroForms (e.g., pdf‑lib or a viewer SDK with forms API). Provide: open → map fields → fill → export/download.

Product polish

Guided onboarding (sign into Google, grant scopes, set profile vault, choose allowlisted domains, set label filter like label:inbox newer_than:1d).

Clear safety rails: “never auto‑submit”, require per‑item user confirmation, domain allowlist, and risk banners for unknown or unsafe links.

Observability: server logs, client error reporting; no PII in logs.

Packaging for Chrome Web Store with privacy policy & “Limited Use” disclosures; production build scripts; versioned releases.

Non‑functional requirements

Security & compliance: Follow Gmail restricted scope “Limited Use” requirements. Never use Gmail data for ads or unrelated purposes. Encrypt tokens server‑side; rotate and revoke; CSRF on OAuth.

Performance: Content script must scan and map typical forms in <150ms after page idle.

Reliability: Backend job to refresh Gmail watch every ~24h; handle missing/expired historyId by fallbacks.

1) Decide on architecture (extension‑only vs. extension + backend)

Recommended: extension + small backend. The backend can process new mail even when the browser is closed and is the right way to meet “daily” processing reliably. Implement Gmail push notifications (users.watch → Pub/Sub) and use users.history.list to fetch only new changes since the last historyId. This is the Gmail‑recommended pattern for change detection.

🧩 Helpful implementation notes (you/your devs)

Gmail watch loop: call users.watch (per user), store the returned historyId, refresh the watch at least daily, and on Pub/Sub notification call users.history.list to enumerate messageAdded/labelsAdded, then fetch just those messages/attachments. Handle empty histories gracefully and backfill if you lose state.

Permissions: start with activeTab, storage, scripting; request site access on click (“This extension can read and change data on this site”).

Queue design: Store only the minimum: message id, subject, sender domain, safe parsed links, attachment metadata. Never persist raw email bodies unless strictly necessary (and disclose if you do).

Edge cases: dynamic frameworks (React/Vue forms), iframes (needs host permissions), anti‑bot widgets (reCAPTCHA), masked inputs; support manual override when mapping fails.
